# Codex Task: 实施 network profile 第二阶段（T2）

你现在要继续实施 `NETWORK_PROFILE_TASK.md` 的 T2：探测与诊断收敛。

## 先阅读
- `NETWORK_PROFILE_REQUIREMENT.md`
- `NETWORK_PROFILE_DESIGN.md`
- `NETWORK_PROFILE_TASK.md`
- 当前已落地的 T1 代码

## 本轮目标
只做 T2，不要扩到 UI 或后续大范围文档重写：

### T2.1 统一 profile 级探测入口
目标：
- 给当前 network profile 增加一个轻量 readiness 判断
- 至少输出：
  - `ready`
  - `degraded`
  - `unavailable`

要求：
- 优先复用现有 onchain `status/probe` 能力与 config 信息
- 不要做新的重型探测系统
- 不要求现在就联网强探测所有东西；可以先基于当前配置与现有能力给出合理 readiness 分类

### T2.2 启动日志 / status 接入 profile 信息
目标：
- 在 status 或相关输出中体现：
  - 当前 profile id
  - readiness
  - 关键原因/诊断摘要

要求：
- 用户能直接看出当前是 `xlayer-recommended` 还是 `evm-custom`
- 用户能知道为什么是 `ready / degraded / unavailable`
- 保持 KISS，不做过度复杂的诊断面板

## 约束
1. **不要扩 scope**
   - 不做完整 UI
   - 不做新的远程探测服务
   - 不新增 ws/x402/paymaster/AA 实现

2. **优先复用现有结构**
   - `config.ts`
   - `server.ts`
   - `onchain status/probe`
   - 已有 runtime/status 输出

3. **保持诊断结果简单可用**
   - 最好有机器可读字段
   - 同时有人类可理解摘要

## 验证
完成后至少执行：
- `npm run build`
- `npm test`

## 输出
完成后请简短说明：
1. readiness 是怎么判断的
2. status 里新增了哪些字段
3. `xlayer-recommended` 与 `evm-custom` 在诊断上有什么差异
4. build/test 结果

# Codex Task: 按 KISS 方案实现 agent-comm 最小复用入口

你现在要直接实现一套 **KISS、轻量、面向复用** 的 agent-comm 最小入口。

## 背景判断
当前 agent-comm v0.1 的底层能力、runtime、测试都已经有了，但“其他用户能不能真正复用”仍然卡在：
- vault / comm wallet 初始化分散
- peer 注册不够集中
- sender 入口不够薄
- 双实例联调路径不够直白

所以现在的目标不是继续扩协议，而是：**把复用动作收敛成极少数明确入口。**

## 实现目标（就按这个做）
请实现最小复用入口，至少包含以下 4 类能力：

### 1. identity 查看入口
用户能直接查看：
- 本机通信地址
- 本机通信公钥
- 当前 chainId
- 当前 walletAlias

### 2. comm wallet 初始化入口
用户能一步完成：
- 生成/恢复通信钱包
- 写入 vault（按 `COMM_WALLET_ALIAS`）
- 返回 address/pubkey

目标：不让用户先自己理解一堆 vault 细节才能启动。

### 3. trusted peer 注册入口
用户能一步注册 trusted peer，至少包括：
- `peerId`
- `walletAddress`
- `pubkey`
- `status=trusted`

### 4. 最薄 sender 入口
至少支持发送：
- `ping`
- `start_discovery`

要求做到：
- 用户不需要自己拼底层模块
- 能直接指定 peer 并发送命令
- 入口越薄越好

## 实现形式建议
你可以在以下形式中选最合适的最薄方案：
- CLI（优先，如果实现最轻）
- 或最小 API
- 或两者组合，但不要做重

## 约束
1. **保持 KISS**
   - 不扩协议
   - 不引入新架构层
   - 不把功能做大

2. **不要扩出 v0.1 边界**
   - 不做 ws
   - 不做 x402
   - 不做 paymaster
   - 不做多链
   - 不做复杂 sender UI

3. **优先复用现有能力**
   - runtime
   - state-store
   - peer-registry
   - tx-sender
   - 现有 API/CLI 结构

4. **要让别人真能用起来**
   - 不是只补一个内部 helper
   - 而是要有可触达入口

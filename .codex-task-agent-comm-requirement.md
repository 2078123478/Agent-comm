# Codex Task: Agent-Comm 实现复盘与 REQUIREMENT.md 重写

你现在要审视 AlphaOS 当前已经实现的整个通信功能（agent-comm / onchain agent communication），目标不是直接改代码，而是**基于当前事实做一次架构与实现复盘**，输出一个新的 `REQUIREMENT.md`。

## 你的目标

请从以下角度系统审视当前通信功能实现：

1. **有哪些地方实现得过重了？**
   - 是否存在为了黑客松演示/未来扩展预埋过多复杂度的问题
   - 是否有模块、抽象、配置、持久化层次过多
   - 是否有“理论完整但实际当前不需要”的设计

2. **有哪些地方可以更简单、更 KISS？**
   - 在不牺牲当前核心目标的前提下，是否可以用更少组件、更少状态、更少配置完成
   - 是否有“一个更简单的方案就够了”但现在用了更重设计的地方
   - 是否存在可以延后实现的能力，而不应进入当前 requirement

3. **哪些地方对其他用户复用时不好配置？**
   - 哪些环境变量、依赖、链上前置条件、钱包/密钥前置条件、RPC/paymaster/x402 等，对普通用户配置门槛过高
   - 哪些配置项语义不清晰、容易误配、默认值不友好
   - 哪些模块虽然本地能跑，但换一个用户/机器/链环境就容易踩坑

4. **要客观严谨，基于事实**
   - 不要先入为主地“为了简化而简化”
   - 不要假设不存在的代码/能力
   - 不要把未来想法当成当前事实
   - 必须明确区分：
     - 已实现的东西
     - 设计文档里写了但未实现的东西
     - 你建议保留的东西
     - 你建议降级/删减/延后的东西

## REQUIREMENT.md 必须包含的结构

### 1. 背景与目标
- 当前通信功能要解决什么核心问题
- 当前 requirement 应该服务于什么阶段（例如：黑客松 demo / 最小可复用能力 / 未来扩展预留）

### 2. 当前事实盘点（非常重要）
- 已实现能力清单
- 部分实现/未完成能力清单
- 仅存在于设计文档但未真正落地的能力

### 3. 复杂度与负担分析
- 哪些地方复杂度偏高
- 复杂度来自哪里（模块数、配置数、状态数、链上假设、密码学前置、支付/x402 预埋等）
- 这些复杂度是否真的服务于当前目标

### 4. 复用与配置门槛分析
- 对其他用户来说最难配置的点
- 哪些配置必须收敛/减少/隐藏
- 哪些默认值或运行模式需要重新设计

### 5. 新的 requirement 建议（核心）
请写成 **Should / Must / Won’t (for now)** 的形式：
- Must：当前阶段必须保留的能力
- Should：推荐保留但可简化实现的能力
- Won’t (for now)：当前阶段不应纳入 requirement、应该延后或删减的能力

### 6. 推荐的 KISS 版本边界
- 如果把 agent-comm 收敛成一个更简单、更容易复用的版本，边界应该是什么
- 最小可用版本包含什么、不包含什么
- 这个版本为什么更适合当前阶段

### 7. 风险与取舍
- 如果简化，会损失什么
- 如果不简化，会继续承担什么成本
- 给出客观 trade-off，而不是单边结论

## 风格要求
- 客观、克制、严谨
- 先陈述事实，再下结论
- 不要写成情绪化批判
- 不要为了“看起来先进”保留不必要复杂度
- 也不要为了“看起来简单”把关键能力误删

## 额外要求
- 如果你认为当前实现里某个复杂模块其实是合理的，请明确写出来，不要因为任务倾向简化就硬砍
- 如果你发现某些设计文档和代码现实不一致，请在 requirement 里明确指出
- 如果你认为需要把当前目标重新定义为“黑客松版”和“通用版”两层 requirement，也可以这样组织，但必须基于事实
